# MCQ 전환 및 시드 데이터 생성 규칙

정리 전 원본 프롬프트를 검토해 중복 제거, 번호 정리, 이 프로젝트에서 실제로
확인된 버그/안전 규칙을 반영했다. 2026-07-08 답-불일치/LaTeX 감사 작업에서
발견한 버그 유형과 겹치는 규칙은 그 근거를 함께 적어둔다.

## 적용 범위

Show that / Prove / Sketch / Construct 등 **선생님 채점이 필요한 문제(`teacher_review`,
`requiresManualGrading`)는 대상에서 제외**. 그 외 답이 하나로 확정되는 단답/계산형
문제만 객관식(가능하면 4지선다)으로 전환한다.

범위가 매우 크다(exam: 제외 약 18,125문제, `tools/audit-state/CHAPTER_MAP.md` 기준).
**한 번에 전체를 돌리지 않는다** — CHAPTER_MAP.md의 챕터 순서를 따라 하나씩,
파일럿(y7-4a 방식: 검증 후 결과 리뷰) → 확대 순서로 진행한다.

## 규칙 1 — 객관식 옵션의 LaTeX 구분자

`options` 배열 안에 수식이 들어갈 때 `$...$` 대신 반드시 `\( ... \)`를 쓴다.

- [AS-IS] `'$5, 6, 7, 8$'`
- [TO-BE] `'\(5, 6, 7, 8\)'`

**근거:** `src/utils/mathPreprocess.js`의 통화기호 보호 로직이 `$<숫자>` 패턴을
특별 취급하기 때문에, 수식용 `$`와 통화용 `$`가 같은 문자열에 섞이면 파싱이
깨질 수 있다 (2026-07-08 `$\$21$` 버그, 28건 발견/수정 — `tools/scripts/fixCurrencyDollarEscapeBug.js`
참고). `\(...\)`를 쓰면 이 충돌 자체가 발생하지 않는다.

## 규칙 2 — solutionSteps 구조 적용

단일 HTML 문자열 `solution` 대신 `solutionSteps: [{ explanation, workingOut, graphData }]`
배열(3~5단계)을 쓴다.

- `explanation`: 그 단계의 수학적 이유(Why)와 목표(What)를 최소 2문장으로. "x를 구한다" 같은
  행위 나열만 하지 않는다. **⚠️ 반드시 영어로 작성한다 — 한국어 절대 금지.** 학생 대면
  텍스트는 전부 영어라는 프로젝트 표준 규칙(질문, 옵션, hint와 동일). 이 md 파일 자체와
  주석은 한국어로 써도 되지만, Firestore에 쓰는 `explanation` 콘텐츠는 예외 없이 영어.
  (2026-07-08 y10-5a 테스트에서 한국어로 잘못 작성해 즉시 재작업한 사고 있었음 — 재발 금지.)
- `workingOut`: 한 단계에 연산 하나만. 이항/약분/양변제곱 등을 한 스텝에 몰아넣지 않고 전부 쪼갠다.
  예외 없이 LaTeX `\( ... \)`로 감싼다.
- `graphData`: 필요 없으면 `null`.
- 첫 스텝: 주어진 조건 정리 + 사용할 공식/정리 계획.
- 마지막 스텝: 답만 던지지 말고, 그 답이 문제 맥락에서 어떤 의미인지(단위, 정의역 부합 등) 마무리.
- **가능한 한 자세히, 단계를 절대 건너뛰지 않는다.** "계산하면 답은 X" 식으로 중간 과정을
  압축하지 말 것 — 이항, 약분, 대입, 양변 연산 등 학생이 손으로 따라 쓸 수 있는 최소
  단위까지 전부 별도 스텝으로 쪼갠다. 스텝이 3개보다 많이 필요하면(예: 6~7단계) 그렇게
  써도 된다 — "3~5단계"는 하한이지 상한이 아니다.

## 규칙 3 — workingOut 내 특수기호 및 인라인 래핑

집합 기호 `{ }` 등은 `\{` `\}`로 이스케이프하고, 전체를 `\( ... \)`로 감싼다.

- [AS-IS] `'\{0, 1, 2, 3, 4, 5\}'` → 렌더링 시 백슬래시가 그대로 깨져 보임
- [TO-BE] `'\(\{0, 1, 2, 3, 4, 5\}\)'` → 정상 렌더링

멀티플초이스 정답을 보기 좋게 하려고 불필요한 수식을 여기 넣지 않는다.

## 규칙 4 — 답 인덱스 무결성 ⚠️ 절대 위반 금지

**근거:** 2026-07-08 감사에서 y11a-7(59건)·y9-1(12건)·y9-8(7건)·y10-1(11건), 총 89건이
정확히 이 실수(정답 인덱스가 실제 정답이 아닌 다른 보기를 가리킴)였다. 반복해서 나온
가장 흔한 버그 유형이다.

작성 절차:
1. `options` 배열을 먼저 확정한다.
2. 정답이 몇 번째(0부터 시작)인지 센다.
3. 그 정수를 `answer`에 적는다 — 문자열(`"0"`)이나 텍스트(`"x = 5"`) 금지, 반드시 정수.
4. 자가검증: `options[answer]`가 실제 정답 텍스트와 일치하는지 눈으로 확인한다.

**정답은 반드시 독립적으로 재계산해서 검증한다 — `solution`/`solutionSteps` 텍스트와
`answer` 인덱스가 서로 일치한다고 해서 정답이라고 믿지 않는다.** 2026-07-08 감사에서
`y10-1a-q3c`("3/16을 %로") 사례를 발견했다 — `answer`는 18.75%로 수학적으로 맞았는데
`solution` 텍스트 자체에 오타(20.63%)가 있었다. 둘이 일치하는지만 보는 자동 검증으로는
이런 "풀이 텍스트 쪽이 틀린" 경우를 못 잡는다 — 직접 계산해서 실제 정답이 맞는지 확인.

## 규칙 5 — 정답 위치 랜덤화

정답이 항상 같은 위치(특히 `options[0]`)에 있으면 안 된다. 4개 보기 전체에 걸쳐
정답 인덱스가 고르게 분포하도록 섞는다.

**근거:** 2026-07-08 감사에서 y7-4a의 거의 모든 문제가 정답을 index 0에 몰아둔 걸
확인했고(오답 생성 스크립트의 습관적 패턴으로 추정), 과거에도 같은 문제로
"Shuffle options for Year 10 Ch3A-D to randomize correct answers" 커밋이 있었다 —
반복해서 나타나는 이슈다. 정답이 한 위치에 몰리면 학생이 문제를 안 풀고 패턴만
학습할 수 있다. 옵션을 섞을 땐 `answer` 인덱스도 반드시 같이 갱신할 것(규칙 4).

## 규칙 6 — 문제 다양성

- 같은 시나리오에 숫자만 바꿔 반복 출제 금지. 매 문제 다른 컨텍스트(로봇, 항공, 농업,
  의료기기, 화학, 야생동물, 부동산 등) 사용, 같은 스토리 재사용 금지.
- 수학 구조도 다양하게: 순방향 계산 / 역산 / 변화율 / 기간 구하기, 단계 비교, 세금+감가
  혼합, 퍼센트 감소율 등.
- 오답 보기는 실제 계산 실수(부호 반전, 반올림 오류, 단리/복리 혼동 등) 기반으로 동적
  생성한다 — 무작위 숫자 아님.

## 규칙 7 — 중첩 LaTeX 구분자 절대 금지

- [오류] `\(\text{Final Expression} = \(m + 3\)\)` — 안쪽 `\)`에서 non-greedy 정규식이
  먼저 매칭을 끊어버려 KaTeX 파싱 에러(빨간 깨진 텍스트) 발생.
- [정상] `\(\text{Final Expression} = m + 3\)` — 바깥 래퍼 하나만.

**근거:** 규칙 1과 동일한 원리의 버그(구분자 안에 구분자가 있으면 조기 종료). 문자열
조립 시 삽입할 변수에 이미 구분자가 포함돼 있는지 반드시 확인하고, 조립 전에 벗겨낸
뒤 바깥쪽에서 한 번만 감싼다.

## 규칙 8 — teacher_review 문제의 줄바꿈 (⚠️ 적용 전 검증 필요)

문장 끝 마침표(`.`) 뒤에 `\n`을 넣어 줄 단위로 끊는다는 규칙인데, 렌더링 파이프라인
전체에서 `\n`이 실제로 줄바꿈으로 처리되는지 2026-07-08 기준 확인 못 했다. 대량 적용
전에 문제 하나로 실제 화면에서 먼저 테스트할 것.

## 이 프로젝트의 안전 규칙 (원본 프롬프트에 빠져있던 것 — 반드시 추가 준수)

원본 프롬프트의 "Firestore Data Sync Safety Rules"는 문제 개수 감소만 체크하는데,
그것만으로는 이 프로젝트의 실제 사고 유형을 못 막는다. CLAUDE.md 기준 다음을 항상 함께 지킨다:

- **`origin: 'teacher'` 문서는 절대 덮어쓰지 않는다.** 개수가 안 줄어도 teacher가 편집한
  내용을 덮어쓰면 안 됨 — 별도 필드 체크가 필요.
- **`questions` 컬렉션에 문서를 추가/삭제/재업로드/이름변경하는 스크립트를 실행한 뒤에는
  반드시 `node tools/scripts/rebuildQuestionIndexes.js <chapterId>`를 실행한다.** 부분
  인덱스는 없는 것보다 나쁘다 (2026-07-06 y5-ar-4 사고: 23개 중 1개만 노출).
  단순 필드 값 수정(답 인덱스 교정 등, 문서 추가/삭제 없음)은 이 재구축이 필요 없다.

## Seed 파일 / 배포 관련 클래리피케이션

- 이 프로젝트는 2026-07-08부로 **새 콘텐츠에 seed 파일(.js)을 강제하지 않기로 했다** —
  Firestore에 직접 읽기/쓰기 + 쓰기 전 검증(KaTeX 재파싱, 중복 보기 체크, 답 인덱스
  체크) 방식으로 간다. 기존 78개 seed 등록 챕터는 그대로 두되 앞으로 유지보수 안 함.
  (`tools/audit-state/PROGRESS.md`의 "seed 파일 필요성" 논의 참고.)
- 문제 데이터만 바뀌는 경우 **Vercel 재배포가 필요 없다** — 앱은 Firestore를 실시간으로
  읽으므로 즉시 반영된다. "배포"는 앱 코드(React 컴포넌트 등)가 바뀔 때만 필요하고, 그때도
  Vercel이지 `firebase deploy`가 아니다.
- 매 수정마다 `isNew: true`를 세팅해 기존 [NEW] 배지 알림이 뜨도록 한다 (`tools/audit-state/PROGRESS.md`
  "NEW question 배지" 섹션 참고). `isNew`는 자동으로 꺼지지 않으니, 실제로 화면에서 그
  문제를 열람하면 `QuestionBankPage.jsx`가 즉시 `isNew: false`로 되돌린다.

## 실행 방식

1. `tools/audit-state/CHAPTER_MAP.md` 순서대로 챕터 하나씩 진행.
2. 각 챕터: 대상 문제 조회(이미 MCQ인 것 제외) → 위 규칙대로 변환안 작성 → KaTeX
   재검증(自) → `origin:'teacher'` 아닌 것만 Firestore에 직접 write → `isNew: true` 포함
   → `tools/audit-state/fix-<chapterId>-log.json`에 `{id, changedFields:{필드:{old,new}}}`
   형식으로 old/new 값 전체 기록 (롤백 가능하도록).
3. CHAPTER_MAP.md의 해당 챕터 체크박스를 `[x]`로 표시.
